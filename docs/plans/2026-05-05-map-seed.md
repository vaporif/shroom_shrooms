**Goal:** Let the player share or replay a map by passing `--seed N` on the CLI and reading the active seed off the HUD.

**Architecture:** Replace the hard-coded `TerrainSeed(42)` with a single `LaunchConfig { seed: u64 }` resource owned by `fungai_core`. The `bin` crate parses `--seed` with clap and inserts the resolved value before plugins run; `init_resource` in `CorePlugin` covers the test path with a default (`420` in debug, OS entropy in release). `terrain_generation` reads `LaunchConfig` instead of `TerrainSeed`, and the HUD turn line gains a `Seed: N` suffix.

**Tech Stack:** Rust 2024, Bevy 0.18, rand 0.10, clap 4 (binary only).

## Execution Strategy

**Subagents.** Single vertical task — the change touches six files but they're trivially small (~30 LOC excluding tests) and form one coherent slice. Parallelism would add coordination cost without saving wall-clock time.

**Pros / cons of the key choices** (carried from the design doc; recorded here so the executor doesn't have to flip back):

- *Single resolved `u64` instead of `Option<u64>` on the resource.* **Pros:** the rest of the codebase reads a concrete value, no unwrap dance. **Cons:** `Default` is side-effecting in release builds. Mitigated because `main.rs` always inserts an explicit value before plugins run, so the `Default` only fires in tests.
- *`rand::rng()` for the random branch.* **Pros:** idiomatic in rand 0.10, OS-entropy backed via `ThreadRng`. **Cons:** `cargo test --release` would draw a fresh seed each run; no release-mode tests exist today, so theoretical only.
- *clap confined to `bin/`.* **Pros:** `--help`, `--seed=N`, `--seed N` all free, easy to extend later. **Cons:** ~250 KB and a few seconds on a clean `bin` build. Acceptable since `bin` is a thin shell.
- *HUD seed appended to the existing turn line.* **Pros:** no new entity, no layout work. **Cons:** the line gets wider; if it overflows on small windows later, split into a separate line then, not now.

```rust
use bevy::prelude::*;
use rand::Rng;

#[derive(Resource, Clone, Debug, Reflect)]
pub struct LaunchConfig {
    pub seed: u64,
}

pub fn default_seed() -> u64 {
    if cfg!(debug_assertions) {
        420
    } else {
        rand::rng().random::<u64>()

}

impl Default for LaunchConfig {
    fn default() -> Self {
        Self {
            seed: default_seed(),
        }
    }
}

mod tests {
    use super::*;

    #[cfg(debug_assertions)]
    #[test]
    fn launch_config_default_seed_is_420_in_debug() {
        assert_eq!(LaunchConfig::default().seed, 420);
    }
}
```

#### Step 1.5 — Resolve the seed in `main()` and insert the resource

- [x] **Step 1.5.1: Update `bin/src/main.rs` to parse args and insert `LaunchConfig`**

```rust
use bevy::prelude::*;
use clap::Parser;
use fungai_core::{LaunchConfig, default_seed};

mod cli;
mod plugins;

use cli::Args;
use plugins::FungaiPlugins;

fn main() {
    let args = Args::parse();
    let seed = args.seed.unwrap_or_else(default_seed);
    App::new()
        .insert_resource(LaunchConfig { seed })
        .add_plugins((DefaultPlugins, FungaiPlugins))
        .run();
}
```

`insert_resource` runs before `add_plugins`, so `CorePlugin::build`'s `init_resource::<LaunchConfig>()` becomes a no-op (Bevy's `init_resource` only inserts if absent).
